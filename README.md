## How Databricks Git folders fit this spike

A Databricks Git folder is a clone of a remote Git repository stored in the Databricks workspace. It is created from **Workspace** > **Create** > **Git folder** (or with `git clone` from the Databricks web terminal), and it tracks the cloned repository's branches and changes. In a workspace, a clone is typically located under a path such as:

```text
/Workspace/Users/<your-email>/<parent-folder>/<git-folder-name>
```

The Git folder is source-control/workspace storage; it is **not** a Python package installation mechanism. Cloning this monorepo makes the checked-out files, notebooks, and `pyproject.toml` files available in the workspace, but it does not install its Poetry projects, build wheels, or automatically expose every nested project's `src/` directory as a Python import root.

Databricks recommends that each collaborator create their own Git folder and work on their own branch. Git operations are performed against that workspace clone: pull updates the checked-out files, while commit and push publish local changes. Pulling upstream changes clears notebook state, so results should not be treated as durable notebook artifacts.

### Sparse checkout

Databricks supports optional **cone-mode sparse checkout** when creating a Git folder. It is a client-side Git setting that limits the clone to selected directory subtrees—useful when a monorepo exceeds Git-folder size limits. A cone pattern such as:

```text
src_style
```

recursively includes the contents of `src_style` and also includes files immediately under the monorepo root. Multiple patterns can be specified. The patterns can later be edited in Git-folder settings, but a sparse checkout created in this mode cannot be converted back to a non-sparse checkout.

Sparse checkout determines **which paths are present** in the clone. It preserves those paths' relative layout: it does not move `src_style/src/src_style` to the Git-folder root, configure `PYTHONPATH`, or otherwise alter Python import resolution.

See Databricks' [Create and manage Git folders](https://docs.databricks.com/aws/en/repos/git-operations-with-repos) documentation for the supported Git workflow, permissions, sparse-checkout settings, and limitations.

## Databricks Git-folder import-path resolution

For Databricks Runtime 11.3 LTS and later, Databricks automatically adds the **Git-folder repository root** to Python's `sys.path`. This makes Python files and packages directly below the cloned repository available for normal imports. On Databricks Runtime 14.0 and later, code run locally also has the directory containing the running notebook or script as its default current working directory (CWD). The CWD is relevant because Python normally searches it too.

For a Git folder cloned to:

```text
/Workspace/Users/<your-email>/<parent-folder>/spike-janus-databricks-sdlc
```

Databricks therefore provides this monorepo root as an import root:

```text
.../spike-janus-databricks-sdlc
```

Python resolves an absolute import by looking for the first package/module component immediately under each import root, then looking for the remaining components beneath that result. It does **not** recursively scan arbitrary subdirectories looking for `src/` directories or `pyproject.toml` files.

Applied to this repository:

```text
spike-janus-databricks-sdlc/     # Databricks-provided import root
├── src_style/                   # project directory
│   └── src/
│       └── src_style/           # import package
└── flat_style/                  # project directory
    └── flat_style/              # import package
```

- `from src_style...` first resolves `src_style/` as the outer project directory. It cannot reach the package in `src_style/src/src_style/` without adding `src_style/src` as another Python import root or installing the project. The longer `src_style.src.src_style...` path is an accidental traversal of repository directories, not the intended package API.
- `from flat_style...` likewise first resolves the outer `flat_style/` project directory. With this project's conventional flat layout, the actual `flat_style/flat_style/` package is one level too deep for an import from the monorepo root. It becomes importable when the **project directory** (`.../spike-janus-databricks-sdlc/flat_style`) is itself an import root.
- A package placed directly at `spike-janus-databricks-sdlc/<package_name>/` can be imported from a notebook in this Git folder, subject to normal Python library-precedence rules.

Databricks can distribute package-structured directories that are on `sys.path` to executors on Runtime 13.3 LTS and later, but that does not add nested project directories to the driver's import path. For the authoritative behavior and version details, see [Work with Python and R modules](https://docs.databricks.com/aws/en/files/workspace-modules).

## Team-imposed import constraints

This spike compares Python project layouts for projects inside a monorepo cloned as a Databricks Git folder.

The following are **constraints imposed by this team for the spike**. They are not Databricks platform restrictions: Databricks supports `%pip`, wheel/library installation, and explicit `sys.path` changes when a team chooses to use them.

- Do not use `%pip install` notebook magic.
- Do not install a `.whl`.
- Do not modify `sys.path` in notebook code.

A shell `PATH` entry only locates executables; it does **not** make Python packages importable. Python imports are resolved from `sys.path` (which can be populated by an installed distribution, the interpreter's working/import root, or external `PYTHONPATH` configuration).

Sparse checkout controls which paths are cloned; it does not change their relative locations or make a project's `src/` directory importable. Because Databricks Git folders add the monorepo root—not every nested project root—to `sys.path`, a `src`-layout project still requires installation or an externally configured Python import root that points to `<project>/src`.

The projects in this repository demonstrate the tradeoff:

- [`src_style`](./src_style/README.md): conventional `src/` layout, preferred for package isolation and distribution correctness, but not directly importable under the constraints above.
- [`flat_style`](./flat_style/README.md): conventional flat layout. From the Databricks-provided monorepo-root import path, its source-tree import is `flat_style.flat_style...`; its shorter Poetry-installed import is `flat_style...`. This illustrates the naming tradeoff of an independently managed flat-layout project inside a monorepo.
